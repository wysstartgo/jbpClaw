# 青数覆盖层总索引

## 1. 目标

这份索引文档用于把当前分支中“相对 `main` 需要刻意保留、且后续应按覆盖层回贴”的内容集中列出来，作为后续从最新 `main` 新拉分支后的总导航。

当前建议坚持的总策略：

1. 先文档化覆盖层。
2. 再从最新 `main` 新拉分支。
3. 最后按覆盖层分域回贴，而不是继续在当前脏分支上做高耦合大 merge。

这样做符合：

1. `KISS`：先把边界讲清楚，再迁移。
2. `YAGNI`：先保留已经明确要保留的域，不提前做大范围重命名或重构。
3. `SOLID`：把公共主干和业务覆盖层分开处理。
4. `DRY`：避免后续每次继续对齐 `main` 时反复人工搜差异。

## 3. 建议迁移顺序

后续从最新 `main` 新起分支后，推荐顺序如下：

1. 先回贴品牌覆盖层，确保产品名、品牌介绍、品牌图形语言和主题主色稳定。
2. 再回贴工作台 UI 与样式覆盖层，恢复双侧栏壳层、工作台结构和主题桥接。
3. 然后回贴登录认证覆盖层，恢复登录入口、认证配置、bridge、deep link 与登录态联动。
4. 再回贴内置治理链覆盖层，恢复 managed catalog、治理预览与只读边界。
5. 再回贴宠物伙伴覆盖层，恢复宠物资源、IPC、浮窗、session 通知、Codex 风格交互和打包资源。
6. 最后回贴唤醒与 TTS 覆盖层，恢复语音输入、唤醒、朗读与相关打包资源。

原因：

1. 品牌和 UI 更偏“外层壳”，先稳定视觉和结构，后续功能域更容易接。
2. 登录认证决定 server models、Portal bridge、登录欢迎浮层和 managed 可见性，应在治理链之前稳定。
3. 治理链和语音链更偏“中层能力”，需要建立在外层壳与认证入口稳定的前提下。

## 4. 四类红线

### 4.1 不要顺手重命名全部 `lobsterai`

这些标识很多已经带有兼容意义，例如：

1. deep link scheme
2. 用户数据目录
3. 数据库文件名
4. 更新与 Portal 链路
5. session key / task key

### 4.2 不要只迁 UI，不迁主题系统

如果只迁组件，不迁以下内容，最终会出现“结构像了，颜色和交互没完全像”的半成品状态：

1. `src/renderer/theme/css/themes.css`
2. `src/renderer/theme/tailwind/plugin.cjs`
3. `src/renderer/index.css`
4. `src/renderer/services/theme.ts`

### 4.3 不要只迁治理 UI，不迁主进程治理主干

如果只迁 renderer 侧的治理预览与 bundle 展示，而漏掉：

1. `src/main/qingshuModules/*`
2. `src/main/qingshuManaged/catalogService.ts`
3. `src/main/main.ts` 中的治理 IPC

那么界面会出现，但实际数据和治理分析不会成立。

### 4.4 不要只迁设置页，不迁语音运行时链路

语音相关最容易只迁 `Settings.tsx`，但真正缺一不可的是：

1. shared constants
2. `preload` IPC 暴露
3. `main.ts` IPC handler
4. `macSpeechService / wakeInputService / ttsRouterService / macTtsService / edgeTtsService`
5. `resources/macos-speech/*`
6. 构建与打包脚本

### 4.5 不要只迁登录入口，不迁认证主干

认证相关最容易只迁 `LoginButton.tsx` 或设置页里的认证配置，但真正缺一不可的是：

1. `src/common/auth.ts`
2. `src/main/auth/*`
3. `src/main/main.ts` 中的 `authWindow / deep link / IPC handler`
4. `src/main/preload.ts`
5. `src/renderer/services/auth.ts`
6. `src/renderer/store/slices/authSlice.ts`
7. `src/renderer/components/LoginWelcomeOverlay.tsx`

### 4.6 不要只迁宠物 UI，不迁 pet 主进程真源

宠物伙伴不是一个单独的 React 挂件。只迁 `PetCompanion.tsx` 会造成浮窗残影、右键关闭按钮裁切、session 消息点击后复活、历史 completed session 冒出等问题。真正缺一不可的是：

1. `src/shared/pet/constants.ts` / `src/shared/pet/types.ts` / `src/shared/pet/config.ts`
2. `src/main/pet/petIpc.ts`
3. `src/main/pet/petWindowController.ts`
4. `src/main/pet/petStore.ts` / `src/main/pet/manifest.ts` / `src/main/pet/catalog*.ts`
5. `src/main/preload.ts` 的 `window.electron.pet` IPC 暴露
6. `src/renderer/pet/petService.ts`
7. `src/renderer/pet/PetCompanion.tsx`
8. `src/renderer/pet/PetSettingsSection.tsx`
9. `src/renderer/App.tsx` 的 `#pet-floating` 与 `app:openPetSession` 承接

## 7. 后续合并保护索引

这一节用于当前 `front-design-merge` 继续选择性合入 `origin/main` 时判断“能否直接合入”。原则是：

1. 不涉及青数覆盖层的公共模块，可以优先按 `origin/main` 直接合入。
2. 只在边界上触碰青数覆盖层的模块，可以合入公共底座，但必须保留青数入口与兼容胶水。
3. 直接承载青数品牌、工作台、内置治理链、登录承接、唤醒/TTS 的模块，不能整包替换，只能按小 diff 手工挑选。

### 7.1 可优先直接合入的公共模块

这些模块本身不属于青数覆盖层，后续可以作为低风险合并白名单，但仍需跑对应测试：

| 模块 | 建议合入方式 | 说明 |
| --- | --- | --- |
| `src/renderer/components/im/*` | 可按文件或目录批量对齐 `origin/main` | IM 设置页本身不是青数覆盖层。 |
| `src/renderer/services/im.ts` | 可直接对齐后补类型适配 | 属于 IM 配置服务，不承载青数品牌或治理链。 |
| `src/renderer/store/slices/imSlice.ts` | 可直接对齐后跑 slice 测试 | 属于 IM 状态底座，不应混入青数业务展示。 |

### 7.4 当前建议执行顺序

后续继续合入 `origin/main` 时，建议按以下顺序：

1. 先合入 IM 白名单模块。
   - 目标：POPO/NIM/多实例 IM UI 与公共网关能力尽量与 `main` 拉齐。
   - 验证：IM store / gateway / renderer IM 设置页测试，外加 Agent 创建/编辑 IM 绑定保存测试。
2. 再合入 OpenClaw lifecycle/runtime 灰区小切片。
   - 目标：启动、重启、runtime patch、日志、配置投影稳定性继续对齐。
   - 验证：OpenClaw engine/config/runtime packaging 测试，检查任务运行中延后重启策略。
3. 再处理 per-agent `modelSlice`。
   - 目标：先做兼容胶水，再决定是否迁完整结构。
   - 验证：server models hydration、Provider 选择、Agent 模型保存与退出登录清理。
4. 最后处理认证流公共能力。
   - 目标：只吸收 `origin/main` 的公共 auth bugfix，不覆盖青数 Portal / QTB / 飞书扫码 / bridge 主干。
   - 验证：登录态恢复、退出登录、server models、managed catalog refresh、欢迎浮层。

### 7.5 IM UI 是否涉及青数覆盖层

结论：IM UI 本体不属于青数覆盖层，可以作为优先合入对象；但它和青数覆盖层有两个交叉点：

1. Agent 创建/编辑页的 IM 绑定入口在 `AgentCreateModal.tsx` 和 `AgentSettingsPanel.tsx` 中，这两个文件同时承载青数内置治理链和 managed Agent 只读语义。
2. 部分 IM 会驱动 managed session / scheduled task / OpenClaw channel session，合入后必须验证“通道实例 -> Agent 绑定 -> 会话/定时任务投递”的链路。

因此推荐做法是：IM 设置页、IM store、IM gateway 可以直接对齐；Agent 页里的 IM 绑定 UI 只抽公共逻辑，不整包替换。

### 7.6 MCP 原生迁移是否涉及青数覆盖层

结论：外部 MCP server 的原生 OpenClaw 配置属于公共能力，青数 managed tools 已完成等价 native MCP endpoint，因此可以按 `origin/main` 的原生 MCP 主干继续合入；但必须保留青数 managed tool invoke 对登录态、catalog snapshot 和治理链的主进程内复用。

当前关键依赖：

1. `src/main/qingshuManaged/managedMcpServer.ts` 在主进程内提供 `qingshu-managed` Streamable HTTP MCP endpoint。
2. `src/main/qingshuManaged/catalogService.ts` 提供 managed tool manifest 和 invoke wrapper，后端调用仍使用当前青数认证态。
3. `src/main/main.ts` 在 OpenClaw 配置同步前启动 AskUser server 与青数 managed MCP server，并把所有 enabled MCP servers 写入 `mcp.servers`。
4. `src/main/libs/mcpBridgeServer.ts` 只负责 AskUser `/askuser`，不再提供 `/mcp/execute`。

推荐做法：

1. 后续 main 分支 MCP 相关更新以 `mcp.servers` 原生路径为准，不再恢复 `McpServerManager`、`/mcp/execute` 或 bridge tool manifest。
2. 可以继续合入 `resolveStdioCommand`、OpenClaw native MCP 热重载、stdio/SSE/HTTP transport 兼容 bugfix。
3. 涉及青数 managed tools 时，必须验证登录成功/退出登录/catalog refresh 后 `qingshu-managed` tools 的暴露与撤销正确。

专项文档：

1. `specs/refactors/mcp-native-migration/2026-05-18-qingshu-managed-tools-native-mcp-design.md`
   - 目标：新增青数 managed tools 原生 Streamable HTTP MCP server。
   - 当前状态：A-E 轮已完成；旧 bridge execute、`McpServerManager`、bridge sync UI / IPC 已移除，AskUser-only server 保留。

## 8. 后续 `main` 更新快速合入 SOP

这一节用于回答“以后 `main` 分支再有更新，如何快速、安全合入”。核心思路是：把青数差异看成覆盖层，把 `main` 看成公共主干；每次更新先判断变更落在哪个层，再决定直接合入、灰区小步挑选，还是覆盖层保护。

### 8.1 推荐工作流

1. 更新基准
   - `git fetch origin main`
   - 切到本地 `main` 并快进到 `origin/main`。
   - 回到当前工作分支或新建合入分支。
2. 生成差异清单
   - 查看 `origin/main` 新增提交。
   - 按路径把变更分成三类：公共白名单、灰区模块、覆盖层红线。
3. 白名单优先合入
   - IM、shared contract、低耦合测试、OpenClaw packaging helper、Provider 纯 helper 等优先直接合入。
   - 每合一组就跑对应测试，不堆大批量改动。
4. 灰区小步挑选
   - `main.ts`、`preload.ts`、`openclawConfigSync.ts`、`openclawEngineManager.ts`、`AgentCreateModal.tsx`、`AgentSettingsPanel.tsx` 只能按 diff 挑公共 bugfix。
   - 保留青数认证、managed catalog、治理 IPC、唤醒/TTS、Agent 治理胶水。
5. 覆盖层红线保护
   - 品牌、工作台 UI、内置治理链、登录承接、唤醒/TTS 相关文件不整包覆盖。
   - 如必须修改，先说明触碰原因，再在对应覆盖层文档中追加变更记录。
6. 验证与打包
   - 跑能力域测试。
   - 跑两套 TypeScript。
   - 跑 `git diff --check`。
   - 如打 `.app`，必须实际启动验证，不只看 electron-builder 成功。

### 8.2 每次 main 更新后的分类命令

建议每次先生成这几份清单：

### 8.3 推荐批次模板

后续每次 `main` 更新，推荐固定分 4 批：

1. A 批：纯公共底座
   - 类型、测试、工具函数、构建脚本、OpenClaw packaging helper。
   - 目标：尽快拿到低耦合 bugfix。
2. B 批：OpenClaw / Provider / 模型配置
   - runtime patch、gateway lifecycle、provider helper、模型配置 helper。
   - 目标：保持公共 AI 引擎能力跟进。
3. C 批：IM / Agent 绑定
   - IM gateway、IM 设置页、shared IM contract、Agent IM binding helper。
   - 目标：保持多实例 IM 与公共主干一致。
4. D 批：覆盖层灰区
   - `main.ts`、`preload.ts`、Agent 创建/编辑、工作台局部接缝。
   - 目标：只吸收必要公共 bugfix，不改变青数产品壳和治理链。

### 8.4 每批最小验收

每批完成后至少做：

1. 对应能力域测试。
2. `npx tsc --project tsconfig.json --noEmit`。
3. `npx tsc --project electron-tsconfig.json --noEmit`。
4. `git diff --check`。
5. 检查本批是否触碰覆盖层红线；如触碰，补对应覆盖层文档。

宠物伙伴覆盖层额外验收：

1. `npm test -- petIpc petService PetCompanion petWindowController`
2. 打开浮窗后右键宠物，左侧只出现小圆形关闭按钮，不能出现大菜单裁切。
3. 有 running / waiting / review / failed session 时，宠物左侧白底黑字 session 列表展示正确。
4. 展开状态只显示右上角向下收起按钮；收起后只显示数量徽标；两者不能同时出现。
5. 点击已完成 session 后，主窗口跳到对应会话，宠物消息列表中该条消失，且不会被主窗口重新投影后复活。

7. 拖动宠物时只移动浮窗位置，不出现文字选区/截图残影，并触发对应拖动动画状态。
8. 设置页宠物管理区展示“唤醒宠物”“刷新宠物”，内置宠物和自定义宠物分开；刷新必须走 `pet:refresh` 主进程 IPC，并重新读取内置、缓存、QingShuClaw 导入与 Codex 自定义宠物。

10. 打 `.app` 后确认 `Contents/Resources/pets` 存在，并真实启动 `release/mac-arm64/QingShuClaw.app`。

### 8.5 打包验收新增硬规则

仅 `electron-builder` 成功不代表 `.app` 可打开。每次打 `.app` 后必须做：

```bash
npx electron-builder --mac dir --arm64 --config electron-builder.json
codesign --force --deep --sign - release/mac-arm64/QingShuClaw.app
codesign --verify --deep --strict --verbose=2 release/mac-arm64/QingShuClaw.app
open -a "$PWD/release/mac-arm64/QingShuClaw.app"
```

同时检查日志：

```bash

```

最小通过标准：

1. 日志出现 `initApp: store initialized`。
2. 日志出现 `OpenClawTokenProxy started` 或明确说明本轮不需要 OpenClaw。
3. 如果包含 OpenClaw runtime，日志出现 gateway ready 或 handshake succeeded。
4. `codesign --verify` 通过。

### 8.6 当前已知打包坑

1. `better-sqlite3` 必须在 Electron main 的 Vite/Rollup external 白名单中。
   - 否则 `.app` 会在 `initStore()` 阶段崩溃。
   - 典型错误是动态 require `app.asar/build/better_sqlite3.node`。
2. 测试 `.app` 可用 ad-hoc 签名修复签名结构。
   - `spctl --assess` rejected 不等于本地不能测试；它只说明未 Developer ID 签名/未公证。
3. optional `moltbot-popo` 可能受 `npm.nie.netease.com` 网络影响缺失。
   - 若本轮测试涉及 POPO，必须先确认 `Contents/Resources/cfmind/third-party-extensions/moltbot-popo` 存在。

### 8.7 是否可以做到快速合入

可以，但前提是坚持这份索引作为“合入路由表”：

1. 公共白名单可以快。
2. 灰区必须小步。
3. 覆盖层红线不能整包覆盖。
4. 每批都要验，尤其是 `.app` 要真实启动。

如果后续 `main` 更新主要落在公共白名单，一般可以 1-2 批完成；如果落在 `main.ts`、OpenClaw 主干、认证主干、工作台 UI，则应拆成 3-5 批，避免一次合入把青数覆盖层打乱。
