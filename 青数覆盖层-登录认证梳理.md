# 青数覆盖层：登录认证梳理

## 1. 背景与目标

这份文档用于把当前分支里“相对 `main` 需要刻意保留”的青数登录认证链路、登录承接 UI、跨端桥接和登录态联动一起抽出来，作为后续从最新 `main` 新拉分支后的回贴清单。

这里关注的是：

1. 登录认证主干
2. 主进程认证适配层与配置
3. renderer 认证服务与状态管理
4. 登录入口、欢迎浮层、设置页认证配置等 UI 承接层
5. Web / Desktop bridge、deep link 与 Portal 跳转
6. 登录态与 `server models`、`qingshuManaged` 的联动

这份文档不重复解释整套平台认证协议细节。协议、接口与会话口径以：

1. `docs/qingshu-auth-bridge-overview.md`
2. `docs/青数飞书登录认证.md`

为事实来源。

## 2. 核心结论

当前分支的登录认证覆盖层不是“一个登录按钮”，而是一组必须整体保留的业务承接链：

1. `src/common/auth.ts` 统一了 renderer / main 共用的认证契约。
2. `src/main/auth/* + src/main/main.ts` 负责桌面端认证适配、token 落地、飞书扫码、bridge 票据、deep link 承接和认证窗口。
3. `src/renderer/services/auth.ts + authSlice.ts` 负责登录态恢复、背景 hydration、退出后本地清理、模型与治理同步。
4. `LoginButton + LoginWelcomeOverlay + Settings` 负责登录前后承接体验，不是可随意拆掉的点状 UI。
5. 顶层 `App.tsx` 同时承接登录欢迎浮层与唤醒浮层，它们都属于当前工作台壳层的一部分。

后续从最新 `main` 新起分支时，这一域必须整组回贴，不能只迁：

1. `LoginButton.tsx`
2. `auth.ts`
3. 某几个 IPC handler

否则容易出现“能打开登录入口，但登录后状态、模型、治理、欢迎浮层或跨端免登断链”的半成品状态。

这套设计也符合当前阶段的工程原则：

1. `KISS`：复用平台用户与平台权限，不再额外造一套桌面账号体系。
2. `YAGNI`：不强行统一所有 token 内部格式，只统一认证语义、桥接能力与落地入口。
3. `SOLID`：把契约、主进程适配层、renderer 服务、UI 承接层拆开，各司其职。
4. `DRY`：共享契约和认证文档作为单一事实来源，避免 Web / Desktop 两侧重复定义口径。

## 10. 登录态生命周期与自动续期

本节用于解释当前登录过期时间、token 存储、自动刷新和 `401` 兜底机制。它是排查“登录后过一段时间接口返回未登录 / 401”的优先事实来源。

### 10.1 token 生命周期口径

当前桌面端以“平台用户 + 桌面双 token 会话”为标准口径：

1. `accessToken`：短时访问 token，默认约 2 小时。
2. `refreshToken`：长时刷新 token，默认约 30 天。
3. refresh 成功后优先保存服务端返回的新 `refreshToken`，因此在服务端支持滚动续期时，连续使用通常不会因为 2 小时到期而直接掉线。

注意：

1. 具体有效期最终以后端签发为准，客户端只按 JWT `exp` 与接口返回结果做判断。
2. 如果同一天早上登录、下午就出现未登录，优先排查“自动刷新是否触发 / refresh 接口是否失败 / 业务体鉴权失败是否被识别”，不优先假设 `refreshToken` 自然过期。

### 10.2 token 本地存储

核心文件：

1. `src/main/main.ts`
2. `src/main/sqliteStore.ts`

主进程通过 `saveAuthTokens(accessToken, refreshToken)` 写入 SQLite kv：

```text
key = auth_tokens
value = { accessToken, refreshToken }
```

读取和清理入口：

1. `getAuthTokens()`
2. `clearAuthTokens()`
3. `clearLocalAuthSession(reason)`

`clearLocalAuthSession(reason)` 不只删除 token，还会：

1. 清除服务端模型元数据。
2. 清除 pending auth callback / bridge code。
3. 通知 renderer：`auth:sessionInvalidated`。
4. 刷新青数 managed MCP runtime 配置，让未登录态能力立即收缩。

### 10.3 refresh 接口顺序

核心文件：

1. `src/main/auth/adapter.ts`

QTB 认证适配层统一通过 `refreshQtbAccessToken()` 刷新 token。当前顺序为：

1. 先调用 `POST /api/qingshu-claw/auth/bridge/refresh`
2. 如果失败，再调用 `POST /api/datachat/qingshu/auth/refresh`
3. 任一接口返回 `code=200` 且包含 `data.accessToken`，即视为刷新成功
4. 成功后调用 `saveAuthTokens(newAccessToken, newRefreshToken || oldRefreshToken)`
5. 两个 refresh 入口都失败时，触发 `onAuthSessionInvalidated('refresh-failed')`

这意味着：

1. 账密登录、飞书扫码登录、Web bridge 到桌面端后的桌面会话，都应统一进入这套 refresh 链路。
2. 如果 refresh 接口失败，客户端会清理本地登录态，而不是继续持有失效 token。

### 10.4 被动自动续期

被动续期是指：业务请求发现 access token 已失效后，自动 refresh 并重试一次。

当前主要有三类入口：

1. `src/main/auth/adapter.ts`
   - `fetchQtbResultWithSession(...)`
   - `fetchCurrentUser(...)`
   - 适用于用户、额度、头像摘要、模型列表、bridge ticket 等认证适配层请求。
2. `src/main/qingshuManaged/catalogService.ts`
   - `fetchWithAuth(...)`
   - 适用于青数内置 Agent / Skill / Tool 的 catalog 与 managed tool invoke。
3. `src/main/qingshuModules/authFetchProvider.ts`
   - `fetchJsonWithAuth(...)`
   - 适用于青数模块化 capability 通过统一认证 provider 发起的请求。

当前识别鉴权失败时，不只看 HTTP `401`，还会识别业务响应体中的登录失效语义：

1. `body.code === 401`
2. `body.code === 403` 且 `body.msg` 包含 `authentication failed`
3. `body.code === 403` 且 `body.msg` 包含 `please login`
4. `body.msg` 包含 `未登录 / 登录失效 / 登录已失效 / 认证失败`
5. `body.msg` 命中 `token expired / token invalid / invalid token`

这里补充 body 识别非常重要。部分后端接口可能返回 HTTP `200`，但业务体实际已经表示未登录；如果客户端只识别裸 `401`，就会出现“下午使用时直接报未登录，但没有自动 refresh”的问题。

### 10.5 主动预刷新

核心文件：

1. `src/main/main.ts`

主进程通过 `setAuthTokensGetter(...)` 给 OpenClaw / 代理链路提供 token。当读取 token 时，会解析 `accessToken` 的 JWT `exp`：

1. 如果距离过期不足 5 分钟，触发 `refreshOnce('proactive')`。
2. `refreshOnce(...)` 会复用当前认证适配器的 `refreshToken()`。
3. 并发 refresh 会通过 `pendingTokenRefresh` 去重，避免滚动 refresh token 被并发消费多次。
4. refresh 成功后会同步 OpenClaw 配置，但不强制重启 gateway。

注意：

1. 这里不是全局定时心跳刷新器。
2. 它更准确地说是“按需触发的预刷新”：只有相关链路读取 token 时才会检查并刷新。

### 10.6 服务器模型与 OpenClaw token 代理续期

核心文件：

1. `src/main/main.ts`
2. `src/main/libs/openclawTokenProxy.ts`
3. `src/main/libs/coworkOpenAICompatProxy.ts`

服务器模型与 OpenClaw 代理侧使用主进程注入的 token getter / refresher：

1. 发起模型请求时获取当前 `accessToken`。
2. 如果上游返回 `401 / 403`，代理链路会调用 refresher 获取新 token。
3. 刷新成功后使用新 token 重试。

这条链路与 managed tool 请求不同：它面向模型代理和 OpenAI 兼容转发；managed tool 请求面向青数内置工具调用。但二者最终都回到同一个主进程认证适配器做 token 刷新。

### 10.7 renderer 登录态恢复与失效处理

核心文件：

1. `src/renderer/services/auth.ts`
2. `src/renderer/store/slices/authSlice.ts`
3. `src/renderer/services/authSessionReset.ts`

应用启动时：

1. renderer 调用 `window.electron.auth.getUser()` 尝试恢复登录态。
2. 成功后进入 `applyAuthenticatedUser(...)`。
3. 背景 hydration 会加载服务端模型、青数 managed catalog、头像摘要等。

认证失效时：

1. 主进程发送 `auth:sessionInvalidated`。
2. renderer 调用 `clearLocalSessionState()`。
3. 本地 UI 回退到未登录态。
4. 青数内置 Agent / Skill / Tool 保留展示，但进入灰态 / 锁定态。
5. 当前会话、active skills、server models 会被清理。

### 10.8 401 排障优先级

如果出现“刚登录不久，使用一会儿后 401 或提示未登录”，按下面顺序排查：

1. 看请求是否经过主进程认证适配层，而不是 renderer 直接裸请求业务接口。
2. 看接口返回是 HTTP `401`，还是 HTTP `200/403` 但 body 中是未登录语义。
3. 看 `refreshQtbAccessToken()` 是否调用了两个 refresh 入口。
4. 看 `/api/qingshu-claw/auth/bridge/refresh` 是否失败。
5. 看 `/api/datachat/qingshu/auth/refresh` 是否失败。
6. 看 `auth_tokens` 中是否真的保存了 `refreshToken`，以及它是否与 `accessToken` 不同。
7. 看是否触发了 `auth:sessionInvalidated`，以及 reason 是否为 `refresh-failed` 或 `qingshu-managed-refresh-failed:*`。

判断口径：

1. 同一天内 access token 过期是正常现象。
2. 同一天内 refreshToken 自然过期通常不正常，除非服务端策略配置更短、refresh token 被覆盖、被登出废弃，或用户登录态被后端主动撤销。
3. 最常见的问题是某条请求没有进入自动 refresh，或业务体未登录没有被识别成需要 refresh。

### 10.9 验收与回归

登录认证变更后至少验证：

1. 账密登录后 `accessToken` 与 `refreshToken` 都落库。
2. 飞书扫码登录后 `accessToken` 与 `refreshToken` 都落库。
3. 手动让 access token 过期后，青数当前用户 / 额度 / 模型请求可 refresh 后重试。
4. 手动让 access token 过期后，青数 managed tool invoke 可 refresh 后重试。
5. 后端返回 HTTP `200` + `code=403` + `msg=please login` 时，客户端会 refresh 后重试。
6. refresh 失败时，客户端清理本地登录态并收缩 managed runtime。
7. 退出登录后 server models 清空，青数内置对象进入锁定态。
